# Contributing to dlt-saga

Thank you for your interest in contributing! This guide covers bug reports,
documentation, features, and plugin development.

## Types of contributions

We welcome:

- **Bug reports** — something not working as documented
- **Documentation** — corrections, examples, new guides
- **Features** — new source pipelines, destination implementations, CLI improvements
- **Plugins** — external packages that register a new pipeline type or destination
- **Tests** — unit, integration, or performance tests

> **Scope note.** New source or destination *types* are best contributed as plugin
> packages (see [Plugin Development](wiki/Plugin-Development.md)) rather than
> as PRs to this repo, unless they are widely applicable and come with tests.

## Development setup

**Prerequisites:** Python 3.11+, [`uv`](https://docs.astral.sh/uv/).

```bash
# 1. Fork and clone
git clone https://github.com/<your-fork>/dlt-saga.git
cd dlt-saga

# 2. Install all dependencies (dev + GCP extras)
uv sync --extra dev

# 3. Install pre-commit hooks (ruff format + lint)
uv run pre-commit install

# 4. Run unit tests to verify your setup
uv run pytest -m unit
```

Local DuckDB-backed integration tests run without any cloud credentials:

```bash
uv run pytest -m integration
```

See [`tests/README.md`](tests/README.md) for details.

## Opening an issue

Search existing issues before opening a new one.

Use the issue templates:

- **Bug report** — version, Python version, OS, destination type, minimal
  reproduction, and full error output.
- **Feature request** — the problem you are solving and your proposed solution
  (or leave the solution open for discussion).
- **Plugin submission** — if you have published a community plugin and want it
  listed in the README.

## Pull request process

1. **Branch** off `main`. Use descriptive names:
   `fix/historize-full-refresh-bug`, `feat/databricks-destination`, `docs/config-reference`.
2. **Keep PRs focused.** One logical change per PR makes review faster.
3. **Write tests** for any new behaviour. Bug fixes should include a regression test.
4. **Update docs** if your change affects user-facing behaviour (CLI flags, config
   keys, output format).
5. **Add a CHANGELOG entry** under `## Unreleased` in `CHANGELOG.md`.
6. **Fill in the PR template** before requesting review.
7. **CI must be green.** All checks (format, lint, type check, unit tests) must
   pass before a PR can be merged.
8. **One approving review** from a maintainer is required.

Commit message style: imperative present tense, 72-character subject line.

- Good: `Fix historize skipping last snapshot when batch size = 1`
- Bad: `fixed bug`, `changes`, `WIP`

## AI-assisted contributions

AI-generated or AI-assisted code is welcome. A few ground rules:

- **A human must own every PR.** The person opening the PR is responsible for its
  content — regardless of whether it was written by them or by a tool they ran.
  "An agent did it" is not a defence against the Code of Conduct.
- **Disclose AI use** in the PR description if the implementation was substantially
  generated by an AI system. This helps reviewers calibrate their attention.
- **No unattended automation.** PRs opened by bots or autonomous agents without a
  human reviewing and approving the submission before it is sent will be closed.
  If you use an agent to draft a PR, read it before clicking Submit.
- **Volume limits.** Multiple near-identical or mass-generated issues/PRs from a
  single actor (human or automated) will be closed as spam and may result in the
  account being blocked.

## Coding standards

Detailed rules live in [`.claude/CLAUDE.md`](.claude/CLAUDE.md). Short version:

- `ruff format` and `ruff check` are enforced by CI — run them before pushing.
- Use type hints on public functions.
- Log with the module-level logger (`logging_manager.get_logger(__name__)`), not `print`.
- Distinguish configuration errors (raise `ValueError`) from unexpected errors
  (let the traceback propagate).
- Do not add code for hypothetical future requirements.

## Plugin development

The plugin system lets you register custom pipeline implementations and
destinations without modifying this repo.

```yaml
# packages.yml (in your consumer project)
packages:
  - namespace: myorg
    path: ./my_pipelines  # Python package with pipeline.py modules
```

Full guide: [wiki/Plugin-Development.md](wiki/Plugin-Development.md).

## Release process

Releases are made by project maintainers:

1. Update `CHANGELOG.md` (move `Unreleased` to a version heading).
2. Bump `version` in `pyproject.toml`.
3. Open and merge a release PR.
4. Push a semver tag (`v0.x.y`) — CI publishes to PyPI automatically.

Community contributors do not need to do anything beyond merging their PR.
